# CLAUDE.md

> 目的: モダンなC言語開発環境における、保守性・可読性の高いコードを書くための規約集。
> 「優れた名前こそ最高のドキュメントである」という考えで「良い物語を語る」コードを目指す。

---

## プロジェクト概要

- **言語**: C11（`-std=c11`）
- **ターゲット**: Linux x86_64
- **ビルドシステム**: make
- **テストフレームワーク**: Google Test（gtest）

---

## ビルド・ツール

```bash
make                 # ビルド
make clean && make   # クリーンビルド
make test            # テスト実行
make lint            # 静的解析（clang-tidy）
make format          # フォーマット（clang-format）
```

### コンパイラフラグ

```makefile
CC      = gcc
CFLAGS  = -std=c11 -Wall -Wextra -Wpedantic -Werror
CFLAGS += -g -O2
CFLAGS += -D_POSIX_C_SOURCE=200809L
```

### テスト

- テストファイルは `test/test_<モジュール名>.cpp` に配置する
- ハードウェア依存部分はモック（`test/mocks/`）を使用する

```cpp
#include <gtest/gtest.h>

extern "C" {
#include "sensor.h"
}

TEST(SensorTest, ReadReturnsValidRange) {
    float val = read_adc(0);
    EXPECT_GE(val, 0.0f);
    EXPECT_LE(val, 3.3f);
}
```

## 設計原則

> コードは「コンパイラが通るか」ではなく「読者が自然に読めるか」を基準に書く。
> コードは書く回数より**読まれる回数の方が圧倒的に多い**。

### 型の安全性
- ビット幅に依存する変数は `<stdint.h>` の固定幅整数型（`uint8_t`, `int32_t` 等）を使う
- フラグ・真偽値は `<stdbool.h>` の `bool` を使い、`int` で代替しない
- バッファ操作は境界チェックのある関数を使う（`snprintf`, `strncpy` 等）

### 変数・スコープ
- 変数は使う直前で宣言し、**必ず初期化**する（未初期化変数は禁止）
- スコープは可能な限り小さく保つ

### エラー処理
- 関数の戻り値を無視しない
- エラーは呼び出し元に伝播させる（もみ消し禁止）
- 失敗時の戻り値は負値（`errno.h` の慣習に従う）

### 関数設計
- 1関数1責任：関数名に `and` が必要になったら分割を検討する
- ポインタ引数は関数先頭で必ず NULL チェックを行う
- 1関数あたり **200行以内** に収める

### メモリ管理
- `malloc` / `free` を使う場合は必ず対になっていることをコメントで明示する

### その他
- マジックナンバーは禁止。すべて名前付き定数（`#define` または `const`）にする
- 新規ファイルを追加した場合は `Makefile` の対象も必ず更新する
- コードを修正したら、関連するコメントも必ず見直す

## 命名規則

> 「優れた名前こそ最高のドキュメントである」

| 対象 | 規則 | 例 |
|---|---|---|
| 変数・関数 | snake_case | `read_sensor`, `buf_size` |
| 定数・マクロ | ALL_CAPS | `MAX_BUF_SIZE`, `TIMEOUT_MS` |
| 構造体タグ | snake_case | `struct sensor_data` |
| typedef 型名 | snake_case + `_t` | `sensor_data_t` |

### 名前の長さ

- 原則 **2〜9文字** を目標とする（第一優先: 意図が伝わること、第二優先: 頻出変数は短く）
- **1文字を許可するケース**: ループカウンタ `i, j, k`、座標 `x, y, z`、汎用 `c, n`
- 10文字以上になる場合の略語: `len`, `buf`, `cnt`, `idx`, `ptr`, `tmp`, `err`, `max`, `min`

```c
// Good
int   retry_cnt;  // リトライ回数
float avg_temp;   // 平均温度

// Bad
int   l;                          // 何の長さか不明
int   the_number_of_retry_count;  // 冗長すぎる
```

## コードスタイル

### 波括弧

`if` / `else` / `for` / `while` の本体が **1行であっても必ず `{}` を使用する**。

```c
if (is_valid) {
    process();
}
```

### ネストの深いコード

- ネストが **4段以上** になる場合は、まず**関数分割でネストを減らせないか**を検討する
- やむを得ず深い場合は、ネストブロックの**直前**に処理全体の意図を説明するコメントを追記する

### 再帰関数

- **再帰関数**になる場合は、ネストブロックの**直前**に**再起関数の意図**を説明するコメントを追記する

### static関数の配置

- **トップダウン**（概要→詳細）: 呼び出す側を上、呼び出される側を下に配置する
- **前方宣言**: ファイル先頭にまとめて宣言し、「関数の目次」として機能させる

```c
// --- 関数宣言（目次） ---
static int fna(void);   // 上位処理
static int fnb(void);   // 中位処理
static int fnc(void);   // 末端処理

// --- 関数定義（概要→詳細の順） ---
static int fna(void) { return fnb(); }
static int fnb(void) { return fnc(); }
static int fnc(void) { return 0; }
```

## コメント規約

> コードは *what* を語る。コメントは *why* を語る。
> この二つが揃って、初めてコードは読者に物語を伝える。

### 基本ルール

- コメントは**日本語**で書く
- Doxygen コメント（`/** */`）を関数・構造体・変数宣言に使用する
- インラインコメントは `//` を使用する
- コメントは**コードの複製ではなく補完**（コードから読み取れない意図・背景・制約を伝える）
- 関数宣言、構造体宣言(構造体と各メンバ)、変数宣言、引数宣言時は**必ずコメントを簡潔に書く**
- 条件分岐(if,switch),ループ(for,while)時は**必ずコメントを簡潔に書く**

```c
// Bad: コードの複製
i++;  // i をインクリメントする

// Good: why を説明
i++;  // 先頭要素はヘッダーのため読み飛ばす
timeout_ms = 200;  // センサーの応答仕様上限が180ms のため余裕を持たせた値
val &= ~0x01;      // LSB はハードウェアの都合で常に不定のため読み捨てる
```

### 区切りコメント

処理のブロックが切り替わる箇所にコメントを置き、読者に「段落」を感じさせる。

```c
// --- 入力値の検証 ---
if (len == 0 || buf == NULL) {
    return ERR_INVALID_ARG;
}

// --- バッファを初期化してデータを読み込む ---
memset(buf, 0, len);
read_sensor(buf, len);
```

### コメントを書くべき場所

| 場面 | 書くべき内容 |
|---|---|
| 変数・関数の宣言 | その意味・役割を日本語で簡潔に |
| 定数 | その値の根拠・出典 |
| アルゴリズムの選択 | なぜこの手法か |
| ネスト4段以上 | 処理全体の意図と構造の説明 |
| 一見奇妙な実装 | 意図的である旨と理由 |

### Doxygen の書き方

```c
/**
 * @brief 注文を処理してデータベースに保存する
 * @param proc 処理対象のorder_processor
 * @param order_id 処理対象の注文ID
 * @return 処理が成功した場合は 0、失敗した場合は -1
 */
int process_order(order_processor_t *proc, int order_id);
```

> 嘘のコメントはコメントがないより有害である。
> コードを修正したら、必ずコメントも一緒に見直すこと。
